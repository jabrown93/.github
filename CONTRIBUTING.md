# Contributing

First off, thanks for taking the time to contribute! ❤️

All types of contributions are encouraged and valued. Please read the relevant section below
before making your contribution. It makes it easier for the maintainer and smooths out the
experience for everyone involved.

> And if you like the project but don't have time to contribute, that's fine. You can still
> support it by starring the repository, sharing it, or referencing it in your own project's
> README.

## Code of Conduct

This project and everyone participating in it is governed by the
[Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this
code. Please report unacceptable behaviour to conduct@jaredbrown.io.

## I Have a Question

Before you ask a question, please search the existing issues to see if it has already been
answered. If you still need clarification:

- Open an issue in this repository's issue tracker.
- Provide as much context as you can about what you're running into.
- Follow the templates provided — it helps the maintainer understand your issue and respond
  faster.

## Reporting Bugs

> You must never report security-related issues or vulnerabilities through the public issue
> tracker. Instead, email security@jaredbrown.io. See [SECURITY.md](SECURITY.md) for
> details.

A good bug report shouldn't leave others needing to chase you for more information. Before
filing a bug:

- Make sure you are using the latest released version.
- Confirm the problem is a genuine bug and not a configuration or environment error on your
  side.
- Search the existing issues to see if it has already been reported.
- Collect the information requested by the bug report form: logs or output, the version you
  are running, your environment, and clear reproduction steps.

Then open an issue using the **Bug Report** form and fill in the requested details. Once
filed, the issue will be triaged, and you may be asked for more information or reproduction
steps before it can be addressed.

## Suggesting Enhancements

Before suggesting an enhancement:

- Make sure you are using the latest version.
- Search the existing issues to see if it has already been suggested. If it has, add a
  comment to the existing issue instead of opening a new one.
- Consider whether your idea fits the scope and aims of the project.

Open an issue using the **Feature Request** form and:

- Use a clear and descriptive title.
- Provide a step-by-step description of the suggested enhancement.
- Describe the current behaviour and explain the behaviour you expected instead, and why.
- Explain why the enhancement would be useful to most users.

## Your First Code Contribution

- Fork the repository and create your branch from the default branch.
- Make your changes, including tests and documentation where appropriate.
- Ensure the linter, formatter, and test suite pass locally.
- Open a pull request and fill in the pull request template.

See the project's README for any project-specific setup instructions.

## Styleguides

### Commit Messages

Commits must follow the
[Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/#summary) standard.
Where configured, this is enforced by CI and pre-commit hooks.

### Code Style

Where a project defines linting and formatting configuration (e.g. ESLint, Prettier), it is
enforced by CI and pre-commit hooks. Please run them locally before opening a pull request.
